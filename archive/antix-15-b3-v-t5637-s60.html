When / how often is use of the the top-level"Update Menu" item advisable or necessary?
(asking because I can't fully recall)
-- after manually installing a program (b/c dpkg post-install and"antiX update menu" hook isn't triggered in that case)
-- after manually creating a .destop file, or editing an existing .desktop file, or manually deleting an existing .desktop file

Because I haven't yet needed to use"Update Menu" while testing this build, but HAVE repeatedly used Menu Manager
(which is currently accessible via Menu --} Applications --} Preferences --} Menu Manager
seems it would be more convenient to have"Menu Manager" at top-level (and relegate"Update Menu" to the Preferences submenu).

As is, the"Menu Manager" launcher is initially displayed under"Personal" submenu (first-run, and until an item is added to"Personal").

problem: it's counter-intuitive to click a launcher pathed under"personal" when user really wishes to amend"applications" submenu

problem (bug): after adding an item to personal, then deciding to remove it, user is presented with a do-nothing item under"personal", labeled"retry".
edit: onHover.... fluxbox menu displays"retry"; JWM menu displays nothing (no flyout for empty submenu); IceWM shows a thin glitchy rectangle
Unless user is aware of the presence of an alternative launcher via controlCentre --} Desktop (tab) --} Edit Menus
(note: the differently labeled name is a further point of potential confusion)
at that point the user is relegated to discovering/using Menu --} Applications --} Preferences --} Menu Manager

wishlist feature: dialogbox displaying static text"just a moment, searching available items..." during the awkward startup delay

wishlist feature: On the"ADD to Applications menu" screen, I would appreciate seeing in advance which category(ies) an item will be added into
