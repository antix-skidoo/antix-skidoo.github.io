There were many changes in the upcoming release of antiX to fix the"can't find antiX filesystem" problem you ran into.

But before you download a pre-release iso and burn a new LiveCD,
========= SCRAPER REMOVED AN EMBEDDED LINK HERE ===========
url was:"https://wiki.ubuntu.com/DebuggingACPI"
linktext was:"this page"
====================================
suggests some boot options you can try. They also suggest you try booting without the"quiet" option in order to see more messages from the kernel. Perhaps the first thing to try would be the"failsafe" boot option. You can get help about other boot options that might help solve your problem by hitting the <F1> key at the Grub boot menu. Then select the"Boot Options" menu option. You might want to write some of them down.

Since your system used to boot and now no longer does, it is very likely the problem is a result of a Linux kernel upgrade. The kernel is constantly evolving to deal with all the changes in PC hardware. Almost always there are boot options available that will allow you to boot older machines. The trick is to find which boot options are needed for your particular machine. It might be worthwhile to Google the model name along with"Linux" to see if someone else has already solved the problem for you. I looked around but I mostly found older posts about getting all the bits of hardware to work after a successful boot.

Congrats on getting Mepis to boot.

I have a couple of ideas for you to try using the test LiveCD you made recently. To be honest, I'm using you as a guinea pig. The problem you are up against is that we are unable to read the /antiX/linuxfs file from the cdrom drive. This file contains an entire Linux filesystem in a compressed form. One solution would be to copy this file to the internal hard drive and then use boot parameters to instruct antiX to use that file.

If you can boot one of your laptops in Windows, then you should be able to mount (or whatever they call it in Windows) the test antiX LiveCD. Make a C:\antiX\ directory and copy \antiX\linuxfs from the cdrom to the C:\antiX\ directory (I'm sorry if I got the Windows lingo wrong).

The next step is to get antiX to look for the linuxfs file on your hard-drive. Unfortunately there is no option on the LiveCD to not look for the LiveCD. I will get this fixed before the release. You can override where we look for the linuxfs file by using the"bdev=xxx" boot parameter. I would suggest trying"bdev=hda1" which is the first partition on the first hard drive.

If you can't boot into Windows, then you might be able to use the Mepis that does boot to do this but that would involve making a custom LiveDVD that is a copy of the Mepis LiveCD with the linuxfs file added to it. This is probably too complicated for now although you should be able to use Mepis to find out which device your internal hard drive shows up as. From the command line:

Code: Select all

$ ls /dev/hd* /dev/sd*
should show you all the internal drives and their partitions. You would have to mount them to see how big each one is. You should also try running the"blkid" command. It will show you all the devices we are interested in and only the devices we are interested in.

You could also use a breakpoint with the test antiX LiveCD to see what devices are available. You can set a breakpoint with"bp=N" where N is one or more digits. The first breakpoint is set with"bp=1". This should get you to a limited shell early in the boot process. From this shell, you can find all your block devices (cdrom, hard drives, usb drives, etc) with these two commands:

Code: Select all

antiX> mdev -s
antiX> blkid
The prompt you will see should be"antiX>". If something shows up on that list that does not start with /dev/sd* or /dev/hd* then this may well be your cdrom device. You can simply try booting with"bdev=xxx" where xxx is device that was listed. For example you might use"bdev=cdrom" or"bdev=/dev/cdrom" (they both do the same thing) if the device /dev/cdrom was listed. Otherwise, use this list to see which device your internal hard drive shows up as.

You can even try to mount your internal drive. Let's say /dev/hda1 shows up on the list. Then use these commands to mount it:

Code: Select all

antiX> mkdir -p /mnt/windows
antiX> mount /dev/hda1 /mnt/windows
antiX> ls /mnt/windows
If that mount command fails, you could also try:

Code: Select all

antiX> mount -t vfat /dev/hda1 /mnt/windows
antiX> mount -t ntfs /dev/hda1 /mnt/windows
If you do mount your internal drive, it would be best to unmount it before moving on:

Code: Select all

antiX> umount /mnt/windows
(Note: no"n" in umount).

Use the"exit" command to continue booting.

I don't know if I have been any help to you but I can tell you for sure that you've been a big help to me in making the next release more friendly for people in your situation. Thank you.
